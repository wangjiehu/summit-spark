# 山巅微光全链路用户体验升级计划

记录时间：2026-06-02  
状态：已审核并执行两包；第一包首屏/设置/清晰度精修已落地，第二包首房微引导/提示互斥已落地

## 1. 总结论

还有明显完善空间。当前版本的训练系统已经很厚，死亡、路线、Drill、mastery 和复盘闭环都开始成形；下一阶段不能继续只堆功能，应该把用户从“打开页面”到“长期复玩”的整条体验线梳顺。

下一轮重点应从“更多提示”转向“更完整的体验节奏”：

- 前期装入：打开页面要快、稳定、有明确状态，首次进入要知道怎么开始、怎么操作、何时计时。
- 新手前 3 分钟：玩家要看懂角色、危险、目标、第一条安全路线和死亡原因。
- 中期练习：重开、Drill、失败演练、路线提示和设置面板要形成低摩擦训练流。
- 后期体验：登顶后不只是看结果，而是进入下一组挑战、专家线、章节目标和长期记录。
- 美感与可读：画面更漂亮，但必须优先服务角色状态、危险物、目标和路线读取。

## 2. 当前基础判断

### 已经具备的优势

- 练习闭环：Focus、Drill、Clean/Pace/Style/Expert、掌握表、路线图、失败演练已经形成可训练结构。
- 路线表达：SAFE/FAST/EXPERT、路线指南针、入场卡和 mastery popup 能解释本房目标。
- 失败反馈：死亡原因、死亡轨迹、死亡点、失败演练让玩家知道下一次怎么改。
- 工程守护：`npm run check`、地图检查、合同检查和文档记录能挡住一部分回退。

### 仍然薄弱的部分

- 装入体验仍像原型：缺少正式 loading、资源就绪状态、失败恢复、首次启动路径。
- 新手引导不够产品化：操作、目标、计时规则、训练系统的入口还依赖玩家自己理解。
- 后期目标还不够强：通关后虽有复盘，但还没有章节感、挑战层、长期档案和可持续目标。
- 美术整体性不够：状态反馈很多，但角色、场景、UI 的统一风格还没有完全建立。
- 浏览器/设备体验还可提升：移动端、触控、手柄、缩放、低性能设备需要更系统的 QA。

## 3. 用户旅程升级框架

### A. 打开页面与装入体验

目标：玩家打开页面后，不应该看到“网页原型正在加载”的感觉，而是进入一个可信的游戏。

应做方向：

- 正式 loading 层：显示游戏名、加载状态、资源准备、首次交互提示。
- 冷启动保护：JS 加载失败、canvas 初始化失败、本地存档读取失败时给出可读 fallback。
- 首次启动路径：第一次进入时显示极简控制提示，只保留开始、移动、跳、冲、抓墙。
- 快速继续：有存档时优先展示“继续练习 / 从头开始 / 打开训练面板”。
- 设置预热：读取并应用 shake、calm effects、ghost opacity、controls preset，避免第一帧闪烁。

验收标准：

- 慢网或刷新时页面有明确状态，不出现空白或半成品 UI。
- 首次玩家 10 秒内知道如何开始。
- 资源失败时页面不会静默坏掉。
- 计时仍然只从真实输入开始。

### B. 前 3 分钟新手体验

目标：不靠长说明，让玩家自然理解“我是谁、去哪、怎么活、怎么变强”。

应做方向：

- 第一房微教程：把移动、跳跃、冲刺、补给、危险物的读法分段呈现。
- 控制提示贴近动作：首次接触墙、首次可冲刺、首次死亡、首次重开时出现短提示。
- 安全线优先：新手默认看到 SAFE 线，FAST/EXPERT 可以作为后续目标浮现。
- 死亡后的第一反馈更短：新手前几次死亡只显示最关键的一句，不让信息过载。
- 第一轮设置入口更明确：训练面板是什么、什么时候该打开，需要用 UI 行为解释。

验收标准：

- 新玩家不用打开 README 也能开始玩。
- 第一次死亡后能理解原因和重试方式。
- 第一房不应该因为信息卡片太多遮挡路线。

### C. 核心跑动体验

目标：每一次跑动都可信、清楚、低干扰。

应做方向：

- 角色状态统一视觉语言：站立、奔跑、跳跃、冲刺、抓墙、Spark、受风、超载、死亡都有一致的状态层级。
- 危险物优先级：尖刺、脆冰、风、棱镜、回声、光继、补给的视觉权重重新审查。
- HUD 信息分层：跑动中只保留速度、房间、目标、死亡、资源；训练数据放入设置/复盘。
- 提示互斥规则：入场卡、死亡 Coach、失败演练、路线卡、Drill HUD 必须有明确优先级，不能一起抢画面。
- 动态可读性 QA：每次新增特效都要在玩家高速移动、ghost 开启、练习线开启时检查。

验收标准：

- 角色在复杂背景上始终可辨认。
- 危险物不会被 ghost、路径线、粒子或 UI 遮挡。
- 失败后提示优先级清楚，不出现三张卡同时解释同一件事。

### D. 训练与复盘体验

目标：训练系统从“功能很多”升级为“下一步自然发生”。

应做方向：

- 训练驾驶舱重排：设置面板第一屏只回答三件事：现在练什么、为什么、点哪里开始。
- Drill 队列连续化：完成或失败后可以一键继续下一项，不必回到表格里找。
- 失败演练升级：死亡点、路线箭头、Drill 建议可以保留到下一次成功或主动关闭。
- 复盘层级：终点复盘先给 3 个行动，再给数据详情，避免玩家被表格压住。
- 训练历史：每房保存最近一次推荐、最近失败原因、最近进步，给长期玩家反馈。

验收标准：

- 从死亡到有效重练不超过一次按键。
- 从通关复盘到下一项 Drill 不超过一次点击。
- 玩家能区分 Clean、Pace、Style、Expert 的目标和失败原因。

### E. 后期体验与长期目标

目标：通关后不结束，而是进入更强的挑战结构。

应做方向：

- 章节/路线完成度：总览显示通关、clean、S、Style、Expert、Flow peak。
- 专家线显性化：每房 Expert 不只是合同，还要有可读的高手路线目标和完成徽章。
- 挑战层：每日/每周挑战、无死亡路线、低死亡路线、全 S、全 Expert。
- Ghost/replay 管理：玩家能选择 PB ghost、target ghost、最近失败 replay 或关闭。
- 长期档案：保存个人最好成绩、练习次数、死亡结构、最近进步、推荐下一目标。

验收标准：

- 通关后有明确的下一目标，不只是“再来一次”。
- 高手玩家能看到值得追的长期指标。
- 练习辅助不破坏真实计时和真实 mastery 数据。

### F. 美感与产品完成度

目标：从“可玩的原型”提升到“有完整气质的游戏 demo”。

应做方向：

- 角色 pass：增强轮廓、朝向、动作姿态和状态色锚点。
- 场景 pass：山巅、冷光、风、棱镜、回声、脆冰形成统一视觉体系，避免单一冷色。
- UI pass：训练面板更像工具界面，复盘更像行动面板，按钮/标签/进度条更统一。
- 动效 pass：loading、开始、死亡、完成、Drill 成功、mastery 提升有一致节奏。
- 音频预留：即使暂不实现，也要为跳、冲、死亡、达成、UI 点击预留事件点。

验收标准：

- 截图一眼能分清角色、危险、目标和背景。
- UI 不像临时调试面板。
- 特效服务状态，不制造假平台或遮挡危险。

## 4. 推荐实施顺序

### 第一包：装入与首次体验

优先级最高，因为它决定玩家是否愿意继续。

任务：

1. 新增正式 loading/ready/start 层。
2. 加首次启动控制提示和快速继续入口。
3. 加初始化失败 fallback。
4. 检查设置应用是否在第一帧前完成。
5. 浏览器验证冷启动、刷新、首次进入、已有存档进入。

### 第二包：提示互斥与新手前 3 分钟

任务：

1. 建立屏幕提示优先级表。
2. 第一房和第一次死亡做短提示。
3. 失败演练、入场卡、路线卡、Drill HUD 做统一互斥规则。
4. 检查移动端和小窗口是否遮挡。

### 第三包：训练驾驶舱重排

任务：

1. 设置面板第一屏重排为“现在练 / 接着练 / 数据详情”。
2. Drill 完成/失败后提供继续下一项。
3. 复盘先给行动卡，再给完整数据表。
4. mastery 路线图和训练计划合并成更清楚的工作流。

### 第四包：后期挑战层

任务：

1. 增加章节完成度总览。
2. 增加全 clean、全 S、全 Expert、Flow peak 目标。
3. 强化 Expert 线徽章和长期档案。
4. 引入可重复挑战结构。

### 第五包：视觉产品化

任务：

1. 角色 silhouette 和动作姿态 pass。
2. 场景色彩和机制视觉层级 pass。
3. UI 组件统一 pass。
4. 动效节奏和音频事件预留。

## 5. 质量门与审查清单

每个实施包完成前必须通过：

- `npm run check`
- `git diff --check`
- 本地浏览器冷启动 smoke
- 本地浏览器失败/重开 smoke
- 小窗口截图检查
- ghost/练习线开启截图检查
- GitHub Pages 线上 marker 检查

体验审查问题：

1. 玩家是否知道现在能做什么？
2. 玩家是否知道为什么死？
3. 玩家是否知道下一次练什么？
4. UI 是否遮挡危险物或角色？
5. 训练辅助是否污染真实计时、clean、PB、mastery？
6. 新增视觉是否提升可读性，而不是只让画面更忙？

## 6. 需要审核的方向取舍

这份计划建议下一轮先做“装入与首次体验”，而不是继续加训练表。原因是训练系统已经领先，但第一印象仍偏原型；如果要全面提高用户体验，入口体验必须先补上。

请重点审核以下取舍：

- 是否同意下一轮先做 loading/ready/first-run，而不是继续堆后期训练功能。
- 是否接受新手前几次死亡减少信息量，把完整教练内容留给训练面板。
- 是否同意短期内继续打磨 10 房垂直切片，而不是马上扩房间数量。
- 是否把视觉产品化排在训练驾驶舱之后，避免先做漂亮但未解决流程的问题。

## 7. 暂不做的事项

为避免下一轮失控，暂不建议立刻做：

- 大量新增房间。
- 完整剧情或大段叙事。
- 在线排行榜或账号系统。
- 复杂关卡编辑器。
- 未经验证的大规模重写。

这些都可以后续做，但必须等装入、首次体验、训练流和后期目标先稳定。

## 9. 2026-06-03 最终执行记录

剩余高价值方向：后期挑战层、角色姿态/场景统一视觉、长期档案和线上反馈循环仍然值得继续做，但这次最后提交优先保证首屏到首房的基础体验更稳。

## 10. 2026-06-03 线上装入补充执行记录

本轮再次审查后补齐线上 freshness 和启动失败兜底：

- CSS/JS 引用改为版本化地址，减少 GitHub Pages 部署成功但玩家仍拿旧资源的情况。
- HTML 增加 build version 标记，用于确认线上页面是否已经更新。

- 竖屏/侧边栏短视口下启动层使用全视口 overlay，并限制 start panel 高度，修复标题和按钮被 16:9 舞台裁剪的问题；训练面板打开时 overlay 降层，避免遮挡设置面板。
- 质量门新增资源版本、build version、启动兜底、noscript 和 ready 标记检查。
